Background

這個系統在解決什麼問題？

業務團隊每天面對數百筆停滯商機，難以判斷哪些值得繼續跟進、該怎麼切入、適合邀請客戶參加哪場活動。 過去這些分析完全依賴人工，耗時且缺乏一致性。

😣 過去的痛點

  • 數百筆停滯商機，業務不知道從哪裡開始
  • 卡關原因不明確，話術千篇一律
  • 活動場次眾多，難以判斷哪場最適合哪位客戶
  • 人工分析耗時，每天要重複同樣的判斷流程

✅ 系統如何解決

  • 每日自動從 CRM 撈取並篩選卡關商機
  • AI 分析每筆商機的卡關原因（4 種類別可多選）
  • 依客戶地區、已參加紀錄推薦最適合的活動場次
  • Gemini AI 為每筆商機生成個人化業務突破話術
How It Works

系統如何運作？

從資料擷取到 AI 輸出，整個流程全自動執行，每天 5–8 分鐘完成 ~220 筆分析。

AI Design

AI 的使用方式

本系統使用 Prompt Engineering 搭配 Google Gemini 2.5 Pro，清楚說明為何不選 RAG 或 Fine-tuning。

為什麼選擇 Prompt Engineering？

每筆商機的所有相關資料（痛點、需求、洞察、互動記錄）都已結構化地存在 CRM 中， 可以直接組進 prompt，不需要向量資料庫或 embedding 搜尋（RAG 無必要）。 商業邏輯規則清晰且會持續演進，Fine-tuning 反而難以快速調整。 Prompt Engineering 讓每次迭代可以在幾分鐘內完成，最符合實際業務需求。

Prompt 結構

區塊內容目的
角色設定 你是資深業務主管，正在協助分析一筆卡關商機 引導模型以業務視角思考
客戶資料 痛點、需求、洞察（L1–L7）、聯絡人姓名 提供個人化的客戶背景
商機狀態 卡關類型、卡關類別、建立天數、經營狀況 描述目前的問題嚴重性
推薦活動 活動名稱、場次時間、信任類型（知道/認同/相信） 提供具體的行動建議依據
任務指令 生成一段個人化的業務突破話術，包含開場、痛點呼應、活動邀約 明確定義輸出格式與要求
Engineering Challenges

解決了哪些技術難題？

實作過程中遇到幾個需要仔細設計才能解決的問題。

Challenge 1

活動場次地區對應

同一個活動在全台多個城市有不同場次，推薦時不能直接用「最早場次」，必須找出客戶所在地區的最近場次。

↓ 解決方案

預先建立 地區場次 dict：每個活動 → 每個地區 → 最近場次日期，推薦時 O(1) 查詢

Challenge 2

快取設計避免重複呼叫 API

220 筆 × 2 次 AI 呼叫 = 440 次 API，若每天重跑費用高且耗時。需要識別哪些商機資料沒有改變。

↓ 解決方案

兩層 JSON 快取（judge_ / sales_），以商機核心屬性的 hash 值為 key，資料不變就直接命中快取

Challenge 3

最後日報日期的正確計算

原始實作以「建立日 ±30 天」為窗口找日報，導致逾期商機（建立 >60 天）抓不到近期日報，造成「近90天有日報」卻被標為「已停止經營」的矛盾。

↓ 解決方案

最後一筆日報日期改為從全部日報取 MAX(日期)，±30 天窗口只用於計算其他互動指標

Challenge 4

多帳號 AI 平行執行

220 筆話術若依序生成需要 30–40 分鐘，遠超過業務人員每日可接受的等待時間。

↓ 解決方案

使用 ThreadPoolExecutor 以 15 個執行緒平行呼叫 Vertex AI，縮短至 5–8 分鐘

Challenge 5

活動類型自動分類

活動對客戶的「信任深度」分三種：知道（認識）、認同（接受）、相信（深度認可），需要自動分類才能在 prompt 中給 AI 正確的邀約脈絡。

↓ 解決方案

關鍵字規則優先判斷，無法確定時呼叫 LLM 分類，結果快取避免重複判斷（覆蓋率 100%）

Challenge 6

逾期卡關的精準篩選

「逾期卡關」需要排除完全沒有行動的 D 級商機，但 D 級的判斷依賴後續計算結果，無法在 SQL 階段直接過濾。

↓ 解決方案

先用 SQL 撈出所有逾期卡關，在 Python 完成等級判斷後，再以 post-filter 移除 D 級（此次移除 43 筆）
